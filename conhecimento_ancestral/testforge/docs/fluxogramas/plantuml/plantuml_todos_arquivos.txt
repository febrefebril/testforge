@startuml testforge_fluxo_do_zero

' ============================================================
' TestForge — Fluxo do Zero
' Gravação Enriquecida + Reconhecimento/Fingerprint
' + Curadoria Multiagente + Self-Healing Determinístico
'
' Objetivo:
' Construir um pipeline melhor partindo da hipótese de que
' a gravação ainda NÃO está concluída.
'
' Base conceitual:
' - Taxonomia de falhas: seletores, timing, contexto, estado,
'   DOM dinâmico e interações especializadas
' - Gravação deve capturar ação + contexto + intenção + evidência
' ============================================================

skinparam backgroundColor #FEFEFE
skinparam ActivityBackgroundColor #F7F9FC
skinparam ActivityBorderColor #4466AA
skinparam PartitionBackgroundColor #EEF2FA
skinparam PartitionBorderColor #3355AA
skinparam NoteBorderColor #999999
skinparam NoteBackgroundColor #FFFFF0
skinparam ArrowColor #4466AA
skinparam DiamondBackgroundColor #FFF9C4
skinparam DiamondBorderColor #AA8844

title **TestForge — Fluxo Proposto do Zero**\nGravação Enriquecida → Evidências → Curadoria → Script Final

start

' ============================================================
' 0. PREPARAÇÃO
' ============================================================
partition "0. Preparação da Sessão" {

  :Receber parâmetros da gravação;
  note right
    Entradas mínimas:
    - nome do caso de teste
    - endpoint inicial
    - navegador/canal
    - viewport
    - modo headed/headless
    - diretório de saída
    - perfil de coleta
  end note

  :Validar entradas obrigatórias;
  if (Entradas válidas?) then (Sim)
    :Criar workspace do caso;
  else (Não)
    :Registrar erro de configuração;
    stop
  endif

  :Criar estrutura inicial de artefatos;
  note right
    output/{slug}/
    - manifest.json
    - recording/
    - evidence/
    - generated/
    - runs/
    - agents/
    - final/
  end note

  :Carregar políticas de gravação;
  note right
    Políticas:
    - não capturar segredos em claro
    - mascarar campos sensíveis
    - limitar tamanho de DOM snapshot
    - evitar sleeps fixos
    - preferir locators semânticos
  end note
}

' ============================================================
' 1. BOOTSTRAP DO NAVEGADOR
' ============================================================
partition "1. Bootstrap do Navegador e Observabilidade" {

  :Inicializar Playwright;
  :Criar browser context;
  :Configurar locale/timezone/viewport;
  :Ativar tracing, screenshots e snapshots;

  :Registrar handlers nativos;
  note right
    Handlers:
    - request
    - response
    - console
    - dialog
    - popup
    - framenavigated
    - frameattached
    - pageerror
  end note

  :Navegar para endpoint inicial;

  if (Aplicação abriu?) then (Sim)
    :Salvar baseline inicial;
  else (Não)
    :Salvar screenshot, console e erro de navegação;
    :Encerrar com falha de ambiente;
    stop
  endif
}

' ============================================================
' 2. RECONHECIMENTO/FINGERPRINT INICIAL
' ============================================================
partition "2. Reconhecimento/Fingerprint Inicial" {

  :Executar fingerprint antes da interação;
  note right
    Detectar:
    - JSF / PrimeFaces
    - jQuery / jQuery UI
    - Angular
    - JSP legado
    - Ionic/Cordova
    - iframes
    - shadow DOM
    - formulários
    - componentes ricos
    - padrões de ID dinâmico
  end note

  :Gerar mapa inicial de tecnologia;
  :Gerar mapa inicial de DOM relevante;
  :Gerar catálogo inicial de componentes;

  if (Tecnologia crítica detectada?) then (Sim)
    :Ativar coletores específicos;
    note right
      Exemplos:
      - JSF: ViewState, PrimeFaces widgets, AJAX
      - jQuery UI: dialogs, datepickers, autocomplete
      - Angular: atributos data-*, mudanças assíncronas
      - JSP legado: XHR manual e timing imprevisível
    end note
  else (Não)
    :Usar perfil genérico de coleta;
  endif

  :Salvar recording/fingerprint_initial.json;
}

' ============================================================
' 3. INSTRUMENTAÇÃO DA PÁGINA
' ============================================================
partition "3. Instrumentação de Gravação" {

  :Injetar recorder JS;
  :Injetar overlay de controle;
  :Ativar MutationObserver com debounce;
  :Ativar coleta de eventos DOM;
  :Ativar coleta de candidatos de locator;
  :Ativar coleta de contexto visual;

  note right
    Para cada elemento interagido:
    - tag
    - id/name/type
    - role/aria-label
    - label associado
    - placeholder
    - texto visível
    - classes relevantes
    - bounding box
    - frame/shadow path
    - candidatos de locator
    - screenshot local
  end note

  :Ativar proteção contra dados sensíveis;
  note right
    Campos do tipo:
    - password
    - token
    - secret
    - cpf/cnpj, se política exigir
    devem ser mascarados ou tratados
    por política específica.
  end note
}

' ============================================================
' 4. LOOP DE GRAVAÇÃO ENRIQUECIDA
' ============================================================
partition "4. Loop de Gravação Enriquecida" {

  repeat
    :Usuário executa uma ação;
    :Capturar evento bruto;
    :Capturar contexto antes da ação;
    :Executar ação no navegador;
    :Capturar contexto depois da ação;

    :Classificar tipo do step;
    note right
      Tipos:
      - navigate
      - click
      - fill
      - select
      - date
      - upload
      - autocomplete
      - dialog
      - popup
      - assertion_hint
      - checkpoint
      - network_hint
      - mutation_hint
    end note

    :Gerar candidatos de locator;
    :Pontuar estabilidade dos locators;

    if (Locator principal parece frágil?) then (Sim)
      :Marcar risco de seletor;
      :Gerar alternativas semânticas;
    else (Não)
      :Registrar locator preferencial;
    endif

    if (Houve atividade AJAX/rede?) then (Sim)
      :Associar rede ao step;
      :Gerar sugestão de espera determinística;
    else (Não)
      :Registrar ausência de dependência de rede;
    endif

    if (Houve mutação relevante no DOM?) then (Sim)
      :Aguardar estabilização observada;
      :Registrar tempo de estabilização;
    else (Não)
      :Prosseguir;
    endif

    if (Step envolve iframe/shadow/popup?) then (Sim)
      :Registrar contexto de escopo;
      :Gerar frame_locator/shadow strategy;
    else (Não)
      :Registrar contexto padrão;
    endif

    if (Step envolve componente especializado?) then (Sim)
      :Acionar detector especializado;
      note right
        Componentes:
        - datepicker
        - autocomplete
        - combobox cascata
        - upload
        - rich text iframe
        - drag and drop
        - modal/dialog
      end note
    else (Não)
      :Registrar step comum;
    endif

    if (Usuário marcou checkpoint/assert?) then (Sim)
      :Capturar assert hint rico;
      :Associar evidências ao assert;
    else (Não)
      :Continuar gravação;
    endif

    :Persistir step incrementalmente;
    :Atualizar fingerprint incremental;

  repeat while (Usuário finalizou gravação?) is (Não)

  :Encerrar overlay e coletores;
}

' ============================================================
' 5. FECHAMENTO DA GRAVAÇÃO
' ============================================================
partition "5. Fechamento e Validação da Gravação" {

  :Coletar eventos pendentes;
  :Salvar session_raw.json;
  :Salvar fingerprint_final.json;
  :Salvar evidências brutas;

  :Validar integridade da gravação;
  note right
    Verificações:
    - há navegação inicial
    - sequência de steps contínua
    - steps têm contexto mínimo
    - locators têm candidatos
    - evidências existem
    - dados sensíveis foram tratados
    - asserts têm alvo verificável
  end note

  if (Gravação íntegra?) then (Sim)
    :Promover para normalização;
  else (Não)
    :Classificar problemas da gravação;

    if (Problemas corrigíveis automaticamente?) then (Sim)
      :Aplicar correção na sessão;
      :Registrar correções;
      :Revalidar gravação;
    else (Não)
      :Gerar relatório de gravação insuficiente;
      :Sugerir nova gravação guiada;
      stop
    endif
  endif
}

' ============================================================
' 6. NORMALIZAÇÃO DOS STEPS
' ============================================================
partition "6. Normalização e Modelo Canônico" {

  :Converter eventos brutos em steps canônicos;
  :Remover ruído;
  :Agrupar eventos redundantes;
  :Preservar checkpoints e network hints;
  :Associar cada step ao fingerprint;

  :Gerar steps_normalized.json;
  :Gerar selector_candidates.json;
  :Gerar recording_summary.md;

  if (Modelo canônico válido?) then (Sim)
    :Seguir para geração do script bruto;
  else (Não)
    :Registrar erro de normalização;
    stop
  endif
}

' ============================================================
' 7. GERAÇÃO DO SCRIPT BRUTO
' ============================================================
partition "7. Geração do Script Bruto v0" {

  :Gerar script_v0.py;
  note right
    Regras:
    - usar Playwright Python
    - preferir locators semânticos
    - evitar XPath absoluto
    - evitar IDs voláteis
    - não usar sleeps fixos
    - inserir waits baseados em sinais
    - manter comentários de rastreabilidade
  end note

  :Gerar mapa step_id -> linha do script;
  :Salvar generated/script_v0.py;
  :Salvar generated/script_metadata.json;
}

' ============================================================
' 8. EXECUÇÃO DIAGNÓSTICA
' ============================================================
partition "8. Execução Diagnóstica Inicial" {

  :Executar script_v0 em modo diagnóstico;
  :Coletar trace, screenshots, DOM, rede e console;
  :Salvar runs/run_001/;

  if (Script passou ponta a ponta?) then (Sim)
    :Promover para estabilização determinística;
  else (Não)
    :Identificar step de falha;
    :Gerar failure_signature.json;
    :Combinar evidências da gravação e execução;
  endif
}

' ============================================================
' 9. ROTEAMENTO MULTIAGENTE
' ============================================================
if (Há falha?) then (Sim)

partition "9. Classificação e Roteamento Multiagente" {

  :Classificar família da falha;
  note right
    Famílias:
    1. Seletores frágeis
    2. Timing e assincronismo
    3. Contexto e escopo
    4. Estado da aplicação
    5. DOM dinâmico
    6. Input/interação especializada
  end note

  if (Seletores frágeis?) then (Sim)
    :SelectorRepairAgent;
  elseif (Timing/assincronismo?) then (Sim)
    :TimingAsyncAgent;
  elseif (Contexto/escopo?) then (Sim)
    :ContextScopeAgent;
  elseif (Estado da aplicação?) then (Sim)
    :StateRecoveryAgent;
  elseif (DOM dinâmico?) then (Sim)
    :DynamicDOMAgent;
  elseif (Interação especializada?) then (Sim)
    :SpecialInteractionAgent;
  else (Não classificada)
    :GeneralDiagnosticAgent;
  endif

  :Montar pacote mínimo de evidências;
  note right
    O agente recebe:
    - erro normalizado
    - step que falhou
    - trecho do script
    - candidatos de locator
    - DOM na gravação
    - DOM na falha
    - screenshot da gravação
    - screenshot da falha
    - rede/console relevantes
    - tentativas anteriores
  end note
}

partition "10. Correção, Validação e Controle de Progresso" {

  :Agente propõe patch;
  :Validar patch estaticamente;
  :Aplicar patch em versão temporária;
  :Executar novamente;

  if (Passou?) then (Sim)
    :Aceitar patch;
    :Criar checkpoint estável;
  elseif (Progrediu?) then (Sim)
    :Aceitar provisoriamente;
    :Continuar diagnóstico;
  elseif (Erro mudou?) then (Sim)
    :Reclassificar nova falha;
    :Roteamento para outro agente;
  elseif (Regrediu?) then (Sim)
    :Rollback para checkpoint anterior;
    :Bloquear estratégia tentada;
  else (Estagnou)
    :Registrar tentativa sem progresso;
    :Tentar estratégia alternativa;
  endif

  if (Limite por família excedido?) then (Sim)
    :Acionar LLM fallback controlado ou revisão manual assistida;
  else (Não)
    :Continuar ciclo de reparo;
  endif
}

else (Não)
endif

' ============================================================
' 11. ESTABILIZAÇÃO DETERMINÍSTICA
' ============================================================
partition "11. Estabilização e Self-Healing Determinístico" {

  :Consolidar patches aceitos;
  :Gerar script_curado.py;

  :Aplicar self-healing determinístico;
  note right
    Estratégias:
    - get_by_role
    - get_by_label
    - get_by_text
    - data-testid/data-test/data-cy
    - label_proximity
    - aria_role_strategy
    - frame_reacquire
    - shadow_pierce
    - network_idle_wait
    - dom_stabilization
    - overlay_dismiss
  end note

  :Remover sleeps frágeis;
  :Padronizar waits;
  :Gerar locators.json;
  :Gerar healing_rules.json;
}

' ============================================================
' 12. MASSA DE TESTE E OTIMIZAÇÃO
' ============================================================
partition "12. Separação de Massa e Otimização" {

  :Extrair dados variáveis do script;
  :Gerar test_data.json;
  :Parametrizar script;
  :Executar múltiplas rodadas limpas;
  :Medir estabilidade e performance;

  if (Execuções consistentes?) then (Sim)
    :Promover para entrega;
  else (Não)
    :Enviar evidências para agente apropriado;
  endif
}

' ============================================================
' 13. ENTREGA
' ============================================================
partition "13. Entrega e Aprendizado" {

  :Gerar script_final.py;
  :Gerar relatório de execução;
  :Gerar relatório de curadoria;
  :Atualizar base de conhecimento;
  :Salvar histórico dos agentes;

  note right
    Artefatos finais:
    - final/script_final.py
    - final/test_data.json
    - final/locators.json
    - final/healing_rules.json
    - final/execution_report.json
    - final/agent_history.json
    - final/fingerprint_summary.json
  end note
}

#LightGreen:Teste entregue: script executável ponta a ponta,\ncom evidências rastreáveis, massa separada e self-healing determinístico;

stop

@enduml

LAYOUT_WITH_LEGEND()

Person(test_engineer, "Test Automation Engineer", "Engenheiro responsável por gravar fluxos, executar diagnósticos, curar falhas e gerar artefatos finais de automação.")

System_Boundary(testforge_system, "TestForge - Runtime Local") {
  Container(cli, "TestForge CLI", "Python CLI", "Ponto de entrada local para comandos record, run, curate e finalize. Carrega configurações, políticas e orquestra o fluxo principal.")

  Container(recorder_engine, "Recorder Engine", "Python", "Coordena a gravação enriquecida: captura eventos humanos, checkpoints, assert hints, contexto técnico e persistência incremental dos steps.")

  Container(fingerprint_engine, "Fingerprint Engine", "Python", "Executa fingerprint antes da primeira interação e atualiza sinais técnicos durante a gravação: tecnologias, frames, shadow DOM, IDs dinâmicos, componentes e riscos.")

  ContainerDb(evidence_repository, "Evidence Repository", "File System / JSON / HTML / PNG / ZIP", "Repositório central de evidências: session_raw, session_normalized, fingerprints, DOM, screenshots, rede, console, storage, frames, selectors, traces e artefatos de execução.")

  Container(script_generator, "Script Generator", "Python", "Gera script_v0.py a partir de steps normalizados, locators candidatos, fingerprint, evidências e estratégias de wait/contexto.")

  Container(diagnostic_executor, "Diagnostic Executor", "Python + Playwright", "Executa script_v0.py ou checkpoints, coleta runs, traces, screenshots, DOM de falha, console/rede e produz dados para diagnóstico.")

  Container(multi_agent_curator, "Multi-Agent Curator", "Python", "Classifica falhas, monta pacote mínimo de evidências, seleciona agentes especializados, coordena patches determinísticos, validação, progresso e rollback.")

  ContainerDb(knowledge_base, "Knowledge Base", "JSONL / JSON", "Armazena failure_signatures.jsonl, resolved_patterns.jsonl e healing_rules.json. Fornece histórico para curadoria, classificação e self-healing determinístico.")

  Container(final_artifact_builder, "Final Artifact Builder", "Python", "Consolida script_final.py, test_data.json, locators.json, healing_rules.json, execution_report.json, agent_history.json e fingerprint_summary.json.")

  Container(optional_llm_gateway, "Optional LLM Gateway", "Azure OpenAI Client / HTTP", "Fallback controlado e opcional. Recebe somente pacote mínimo de evidências mascarado e retorna hipótese ou patch candidato, nunca aplicado diretamente.", $tags="Optional")

  Container(local_browser, "Local Browser via Playwright", "Chrome / Edge", "Navegador local controlado pelo Playwright. Executa a aplicação alvo com Recorder JS, Overlay e MutationObserver injetados.")
}

System_Ext(target_web_app, "Target Web Application", "Aplicação web alvo testada pelo TestForge. Pode conter JSF/PrimeFaces, jQuery/jQuery UI, Angular, JSP legado, Ionic/Cordova ou outras tecnologias.")

Rel(test_engineer, cli, "Usa", "CLI local")

Rel(cli, recorder_engine, "Aciona gravação enriquecida", "record")
Rel(cli, diagnostic_executor, "Aciona execução diagnóstica", "run")
Rel(cli, multi_agent_curator, "Aciona curadoria", "curate")
Rel(cli, final_artifact_builder, "Aciona consolidação final", "finalize")

Rel(recorder_engine, local_browser, "Controla e instrumenta", "Playwright API")
Rel(local_browser, target_web_app, "Interage com", "HTTP / HTTPS / XHR / fetch / WebSocket")

Rel(recorder_engine, fingerprint_engine, "Envia página/contexto para fingerprint", "DOM, scripts, frames, shadow DOM")
Rel(recorder_engine, evidence_repository, "Persiste steps e evidências incrementais", "JSON / HTML / PNG / ZIP")
Rel(fingerprint_engine, evidence_repository, "Persiste fingerprints e sinais tecnológicos", "fingerprint_initial.json / fingerprint_final.json")

Rel(script_generator, evidence_repository, "Consome sessão normalizada, locators e evidências", "session_normalized.json / selector_candidates.json")
Rel(script_generator, diagnostic_executor, "Entrega script gerado", "script_v0.py")
Rel(diagnostic_executor, evidence_repository, "Persiste runs e evidências de falha", "execution.json / trace.zip / failure artifacts")

Rel(diagnostic_executor, multi_agent_curator, "Envia falhas diagnosticadas", "failure_signature.json")
Rel(multi_agent_curator, evidence_repository, "Consome evidências originais da gravação e execução", "recording / evidence / generated / runs")
Rel(multi_agent_curator, knowledge_base, "Consulta e atualiza padrões", "failure signatures / resolved patterns / healing rules")
Rel(multi_agent_curator, diagnostic_executor, "Solicita reexecução de patch/checkpoint", "temporary script / checkpoint")

Rel(multi_agent_curator, optional_llm_gateway, "Usa somente como fallback controlado", "pacote mínimo mascarado", $tags="Optional")
Rel(optional_llm_gateway, multi_agent_curator, "Retorna hipótese ou patch candidato", "sugestão não aplicada diretamente", $tags="Optional")

Rel(final_artifact_builder, evidence_repository, "Lê artefatos estabilizados", "generated / runs / agents / final")
Rel(final_artifact_builder, knowledge_base, "Consolida regras determinísticas aprendidas", "healing_rules.json")
Rel(final_artifact_builder, cli, "Disponibiliza entrega final", "paths e relatórios")

UpdateElementStyle(evidence_repository, $bgColor="#FFF2CC", $borderColor="#D6B656")

  Evidence Repository é central.
  A curadoria não deve ignorar dados da gravação:
  session_raw, session_normalized, fingerprints,

note right of EVIDENCE_PACKAGED
  O pacote de evidências deve reutilizar a gravação
  e os artefatos originais da sessão:

  - session_normalized.json

  A curadoria não pode ignorar o contexto coletado
  durante a interação humana.

note right of REGRESSED
  REGRESSED significa que o patch piorou o fluxo,
  quebrou steps previamente válidos ou removeu
  comportamento necessário.

  Neste estado o rollback é obrigatório antes de
  qualquer nova tentativa.
end note

note right of ERROR_CHANGED
  ERROR_CHANGED significa que a assinatura da falha
  mudou após uma tentativa.

  A falha deve voltar para CLASSIFIED para que a nova
  família seja identificada e roteada para o agente correto.
end note

note right of ESCALATED_TO_LLM
  O LLM é fallback controlado.

  Ele só é acionado depois que as estratégias
  determinísticas aplicáveis foram tentadas ou esgotadas.
  Sua saída é apenas hipótese ou patch candidato e deve
  passar por PATCH_VALIDATING antes de qualquer execução.
end note

note right of RESOLVED
  RESOLVED exige evidência de validação:
  - patch aceito;
  - execução comparativa bem-sucedida;
  - checkpoint criado;
  - padrão aprendido registrado na KnowledgeBase;
  - regra candidata de healing rastreável.
end note

  endif

  else (não)

# Revisão crítica dos diagramas PlantUML do TestForge

## 1. Consistência arquitetural

### Inconsistências encontradas

1. **Nome do pedido vs. conteúdo gerado no último diagrama**  
   O pedido menciona “Diagrama de sequência do selfhealing”, mas o texto do requisito descreve novamente um **diagrama de implantação**. O diagrama gerado seguiu corretamente o conteúdo solicitado, porém o nome do artefato ficou semanticamente ambíguo.

2. **Componentes ainda não uniformizados entre todos os diagramas**  
   Alguns diagramas usam `FailureSignatureBuilder`, outros enfatizam apenas `FailureDiagnostic` e `FailureClassifier`. O modelo de domínio possui `FailureSignature`, mas o pacote possui `failure_signature`. Isso é aceitável por camada, mas deve ser padronizado no documento de arquitetura para evitar dúvidas entre classe, módulo e serviço.

3. **Mistura pontual entre runtime e domínio no diagrama de implantação/self-healing**  
   O diagrama de implantação inclui muitos componentes internos de curadoria dentro do nó `Ambiente Python`. Isso ajuda na visualização, mas pode poluir o diagrama de implantação. Para documentação executiva, mantenha só containers/processos. Para implementação, o nível detalhado é útil.

4. **Uso de elementos auxiliares escondidos em alguns diagramas**  
   Foram usados elementos como `TargetBackendAlias`, `edge_hidden_1` e `actions_hidden_1` apenas para layout. Esses elementos podem aparecer ou gerar ruído dependendo do renderizador PlantUML. Recomenda-se remover.

5. **Dependências de pacote potencialmente excessivas**  
   No diagrama de pacotes, `patch` depende de `executor`, `failure`, `knowledge` e `models`; `agents` também depende de `patch`, `knowledge`, `failure` e `evidence`. Isso é coerente, mas exige disciplina para evitar ciclo indireto. A regra recomendada é: `agents` orquestra, `patch` valida/aplica, `executor` executa; `patch` não deve chamar `agents`.

### Ajustes recomendados

- Criar uma tabela de mapeamento entre **Container**, **Package**, **Component** e **Class**.
- Remover aliases escondidos usados só para layout.
- Separar o diagrama de implantação em duas versões:
  - **Implantação resumida**, com nós/processos.
  - **Implantação detalhada**, com componentes internos.
- Criar um diagrama específico de sequência do self-healing, porque o último pedido tinha título de sequência, mas requisitos de implantação.

---

## 2. Aderência ao fluxo do zero

### Pontos fortes

- A gravação foi tratada como **pipeline de coleta de evidências**, não só captura de evento.
- O fingerprint inicial aparece antes da primeira interação.
- O loop de gravação persiste steps incrementalmente.
- A gravação é validada antes da normalização e antes da geração do script.
- Há distinção clara entre `session_raw.json`, `session_normalized.json`, `steps_normalized.json`, `selector_candidates.json` e `script_v0.py`.

### Ajustes recomendados

- No diagrama de atividade, a transição da fase 12 que usa `detach` pode ficar ambígua. Melhor apontar explicitamente de volta à fase de classificação/curadoria ou usar nota indicando retorno lógico.
- Em todos os diagramas, reforçar que `fingerprint_initial.json` é salvo **antes** de `JSInjector` e `RecorderOverlay` quando a intenção for capturar o estado original da página sem instrumentação.

---

## 3. Aderência à taxonomia de falhas

### Pontos fortes

- As seis famílias aparecem no roteamento:
  - seletores frágeis;
  - timing/assincronismo;
  - contexto/escopo;
  - estado da aplicação;
  - DOM dinâmico;
  - interação especializada.
- Existem agentes especializados por família.
- O `ProgressDetector` contém os estados esperados:
  - `PASSED_STEP`;
  - `PROGRESSED`;
  - `STAGNATED`;
  - `REGRESSED`;
  - `ERROR_CHANGED`;
  - `UNRESOLVED`.
- Rollback em regressão está representado.
- `ERROR_CHANGED` retorna para classificação.

### Ajustes recomendados

- Incluir `GeneralDiagnosticAgent` no diagrama de sequência de curadoria como agente para falha desconhecida. Ele aparece no contexto e no diagrama de componentes, mas no diagrama de sequência foi representado apenas como estratégia geral interna do `AgentOrchestrator`.
- Representar explicitamente que `UNRESOLVED` pode voltar para `AGENT_SELECTED` enquanto ainda houver estratégia determinística disponível.

---

## 4. Uso correto de LLM

### Pontos fortes

- O LLM aparece como fallback opcional/controlado.
- Há validação posterior via `PatchValidator`.
- Há proteção via `SensitiveDataGuard` antes de envio externo.
- A resposta do LLM é tratada como hipótese ou patch candidato, não como correção automática.

### Ajustes recomendados

- Incluir em todos os diagramas onde houver LLM a frase arquitetural: **“LLM nunca aplica patch diretamente.”**
- Mostrar que o pacote enviado ao LLM deve conter apenas o mínimo necessário e deve ser mascarado/tokenizado.

---

## 5. Artefatos

### Artefatos cobertos nos diagramas

Os diagramas mencionam ou materializam os principais artefatos:

- `session_raw.json`;
- `fingerprint_initial.json`;
- `fingerprint_final.json`;
- `steps_normalized.json`;

### Ajustes recomendados

- Incluir explicitamente `sensitive_data_report.json` no C4 Container e no diagrama de implantação, pois é essencial para governança.
- Incluir `agent_history.json` no C4 Container como saída do `Final Artifact Builder`.
- Incluir `recording_summary.md` como relatório parcial da gravação.

## 6. Implementabilidade

### Pontos fortes

- Os componentes mapeiam bem para módulos Python.
- O diagrama de pacotes oferece uma organização inicial implementável.
- O modelo de domínio é suficiente para um protótipo incremental.
- Há separação entre gravação, evidência, geração, execução, falha, curadoria, patch, self-healing, dados e entrega.

### Ajustes recomendados

- Criar contratos Pydantic/dataclasses para:
  - `RecordingSession`;
  - `RecordedStep`;
  - `LocatorCandidate`;
  - `EvidenceBundle`;
  - `FailureSignature`;
  - `AgentAttempt`;
  - `Patch`;
  - `HealingRule`.
- Criar interfaces explícitas para agentes:
  - `AgentInput`;
  - `AgentOutput`;
  - `PatchCandidate`;
  - `ProgressResult`.
- Criar camada de porta/adaptador para Playwright, evitando espalhar chamadas diretas ao Playwright pelos agentes.

# Priorização dos diagramas para implementação

## Prioridade 1 — Fonte de verdade para o esqueleto do projeto

1. **Diagrama de pacotes**  
   Deve ser implementado primeiro porque define a organização do código Python e reduz risco de acoplamento circular.

2. **Diagrama de classes / modelo de domínio**  
   Deve vir logo depois para formalizar os contratos dos JSONs e dos objetos internos.

3. **Diagrama de atividade completo**  
   Serve como roteiro macro para backlog, épicos e milestones do protótipo.

## Prioridade 2 — Fluxos críticos executáveis

4. **Diagrama de sequência da gravação enriquecida**  
   Orienta a implementação do `record` e da persistência incremental.

5. **Diagrama de sequência da curadoria multiagente**  
   Orienta a implementação do `curate`, dos agentes e do loop patch/execução/progresso.

6. **Diagramas de estados da RecordingSession e FailureSignature**  
   Devem guiar validações, persistência de status e retomada segura do processo.

## Prioridade 3 — Operação local e visão executiva

7. **Diagrama de implantação runtime local**  
   Útil para explicar ambiente, dependências locais, arquivos e integração com Playwright.

8. **C4 Container**  
   Ideal para comunicação de arquitetura com pessoas que não precisam ver todos os detalhes UML.

9. **Diagrama detalhado de self-healing/implantação**  
   Útil depois que o loop de curadoria estiver prototipado, para consolidar runtime, regras finais e fallback controlado.

    SmartRecorder -> FingerprintEngine : atualizar fingerprint incremental\ncom mutações, tecnologias e componentes observados
    activate FingerprintEngine
    FingerprintEngine -> TechnologyDetector : reavaliar sinais relevantes se necessário
    activate TechnologyDetector
    TechnologyDetector --> FingerprintEngine : riscos incrementais atualizados
    deactivate TechnologyDetector
    FingerprintEngine --> SmartRecorder : fingerprint incremental atualizado
    deactivate FingerprintEngine

    SmartRecorder -> SmartRecorder : criar step enriquecido\nação + contexto + evidência + locators + riscos + hints

    SmartRecorder -> SessionSerializer : persistir step incrementalmente\nsession_raw.json.partial
    activate SessionSerializer
    SessionSerializer --> SmartRecorder : step persistido
    deactivate SessionSerializer
  end

  Usuario -> RecorderOverlay : finalizar gravação
  activate RecorderOverlay
  RecorderOverlay --> SmartRecorder : sinal de encerramento
  deactivate RecorderOverlay

  SmartRecorder -> EventListener : coletar eventos pendentes
  activate EventListener
  EventListener --> SmartRecorder : buffers finais de rede, console, frame e erros
  deactivate EventListener

  SmartRecorder -> EvidenceCollector : coletar evidências finais\nDOM final, screenshot final, storage, frames e console
  activate EvidenceCollector
  EvidenceCollector --> SmartRecorder : evidências finais coletadas
  deactivate EvidenceCollector

  SmartRecorder -> FingerprintEngine : executar fingerprint final
  activate FingerprintEngine
  FingerprintEngine -> Page : inspecionar estado final da aplicação
  activate Page
  Page --> FingerprintEngine : sinais finais
  deactivate Page
  FingerprintEngine -> SessionSerializer : salvar fingerprint_final.json
  activate SessionSerializer
  SessionSerializer --> FingerprintEngine : fingerprint final persistido
  deactivate SessionSerializer
  FingerprintEngine --> SmartRecorder : fingerprint final disponível
  deactivate FingerprintEngine

  SmartRecorder -> SessionSerializer : salvar session_raw.json\ne consolidar recording_summary.md
  activate SessionSerializer
  SessionSerializer --> SmartRecorder : sessão bruta consolidada
  deactivate SessionSerializer

  SmartRecorder -> RecordingValidator : validar integridade da gravação
  activate RecordingValidator
  RecordingValidator -> SessionSerializer : ler session_raw.json, fingerprint_initial.json\ne fingerprint_final.json
  activate SessionSerializer
  SessionSerializer --> RecordingValidator : artefatos de gravação
  deactivate SessionSerializer

  RecordingValidator -> RecordingValidator : verificar steps, timestamps, evidências,\nlocators, checkpoints, fingerprints e mascaramento

  alt Gravação válida e suficiente
    RecordingValidator --> SmartRecorder : gravação aprovada
    SmartRecorder -> SessionSerializer : marcar sessão como pronta para normalização
    activate SessionSerializer
    SessionSerializer --> SmartRecorder : status atualizado
    deactivate SessionSerializer
    SmartRecorder --> CLIRecord : gravação promovida para normalização
  else Gravação inválida ou insuficiente
    RecordingValidator -> SessionSerializer : gerar relatório de gravação insuficiente
    activate SessionSerializer
    SessionSerializer --> RecordingValidator : relatório salvo em recording_summary.md
    deactivate SessionSerializer
    RecordingValidator --> SmartRecorder : gravação rejeitada com evidências do motivo
    SmartRecorder --> CLIRecord : gravação insuficiente para normalização
  end

  deactivate RecordingValidator
  deactivate SmartRecorder
  deactivate BrowserContextManager
  CLIRecord --> Usuario : exibir resultado da gravação
  deactivate CLIRecord

else Entradas inválidas
  InputValidator --> CLIRecord : erros de validação
  deactivate InputValidator
  CLIRecord --> Usuario : exibir erro e não iniciar navegador
  deactivate CLIRecord
end

note over SessionSerializer,RecordingValidator
  A gravação somente é promovida para normalização
  quando existe evidência suficiente para reconstruir:
  intenção, contexto técnico, estado da aplicação,
  locators alternativos e riscos por tecnologia.
end note

@enduml
